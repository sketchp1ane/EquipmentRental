## 2. 开场白建议

> 建议时长：**1～2 分钟**

参考话术：

> "本系统针对建筑行业设备租赁管理中存在的台账分散、流程不透明、安全记录难追溯等痛点，基于 ASP.NET Core 8 MVC 框架设计开发。系统覆盖设备从入库登记到退场评价的完整生命周期，共七个核心业务模块，支持六种角色的差异化权限管控。接下来我将按照真实业务链路，逐步演示各模块的核心功能。"

**可提及的技术亮点（按需选用）：**
- RBAC 权限体系，6 个角色，精细到页面级和操作级
- 进场核验基于唯一核验码（GUID），三日有效期，防止滥用
- 安全交底支持富文本内容 + 电子签名 + PDF 导出
- 文件上传五层安全校验（扩展名、MIME 类型、魔数、大小、路径）
- 报表模块含趋势图、设备利用率、安全统计，支持 Excel 导出

## 4. 分步演示操作

### Step 1 系统登录与角色切换

**目标**：展示 RBAC 权限体系，不同角色看到不同菜单。

1. 用 `admin@equiprental.com` 登录，打开**用户管理**页面。
2. 向评委展示用户列表：5 个演示账号各自对应不同角色。
3. 点击某个用户的"编辑"，展示角色分配界面。
4. 退出，换用 `demo.safetyofficer@equiprental.com` 登录。
5. 指出顶部导航栏中**没有**设备管理、调度管理等菜单——权限控制生效。

**强调点**：菜单项、操作按钮、Controller Action 三层均做了角色鉴权，非授权访问返回 403。

### Step 2 设备入库登记

**登录账号**：`demo.deviceadmin@equiprental.com`

**目标**：展示设备台账模块，演示新建设备流程。

1. 进入**设备管理 → 设备台账**，展示现有 5 台演示设备：
   - 筛选状态下拉框，展示"空闲/出租/维修/待审/报废"全部覆盖。
   - 点击 `DEMO-EQ-001`（塔式起重机），展示详情页：图文信息、历史租赁记录。

2. 点击**新建设备**，填写表单：
   - 设备编号：`TEST-EQ-099`
   - 设备名称：`演示用挖掘机`
   - 分类：`土石方机械 > 挖掘机`（展示两级下拉）
   - 品牌型号、出厂日期、技术参数等按需填写

3. 提交后，设备状态为**待审核**，进入证件管理：
   - 添加一条**产品合格证**（填写证书编号、有效期）

**强调点**：新设备默认"待审核"，只有审核通过后才能被调度，体现业务约束。

### Step 4 线上调度与合同

**登录账号**：先用项目负责人，再切换调度员

**目标**：展示调度申请 → 排期 → 合同生成全链路。

**4b. 调度员排期**（`demo.dispatcher@equiprental.com`）

1. 进入**调度管理 → 待处理申请**，找到刚才的申请。

**4c. 展示合同功能**

1. 进入合同详情，展示自动填充的合同内容。
2. 点击**导出 PDF** 按钮，展示生成的合同 PDF。

**强调点**：
- 调度员选设备时，系统自动排除不可用设备（出租中、资质过期、时间冲突）。

**登录账号**：`demo.deviceadmin@equiprental.com`

**目标**：展示设备在租期间的巡检记录功能。

1. 进入**现场管理 → 故障上报**，展示链路 C 已存在一条**处理中**的故障报告：

## 5. 重点功能亮点说明

答辩时可根据评委提问，展开以下亮点：

### 5.1 安全性设计

| 措施 | 实现方式 |
|---|---|
| CSRF 防护 | 所有 POST/PUT/DELETE 请求强制校验 `AntiForgeryToken` |
| SQL 注入防护 | 全程使用 EF Core LINQ，禁止拼接 SQL |
| 文件上传安全 | 扩展名白名单 + MIME 类型 + 文件魔数五层校验 |
| 富文本 XSS 防护 | 入库前经 `HtmlSanitizer` 过滤危险标签 |
| 密码安全 | BCrypt 哈希存储，禁止明文 |
| 越权访问防护 | Controller/Action 级 `[Authorize(Roles=...)]` 注解 |

### 5.2 业务完整性约束

- 设备状态是严格的状态机：`待审 → 空闲 → 出租 → 维修 → 报废`，非法跳转被服务层拦截。
- 进场核验码三日有效期，避免提前核验后设备状态不一致。
- 资质过期的设备无法被调度员选中（查询层过滤）。
- 押金退款金额服务端计算，不接受前端传值。

### 5.3 文件管理

上传文件保存在 `Uploads/` 目录（不在 `wwwroot`），通过 `FilesController` 鉴权后下发，防止未授权直接访问静态文件。

### 5.4 可扩展性

- 分层架构（Controller / Service / Entity / ViewModel）职责清晰，新增模块只需扩展 Service 层。
- EF Core Code First，数据库迁移版本化管理，支持增量升级。

## 6. 常见答辩问题参考

**Q：为什么选择 ASP.NET Core MVC 而不是前后端分离？**

> MVC 架构更适合这类企业内部管理系统：服务端渲染 SEO 友好、开发周期短、与 ASP.NET Core Identity 集成无缝、表单校验可复用 DataAnnotations。前后端分离适合高并发的 C 端产品，对本项目场景属于过度设计。

**Q：如何保证多用户并发操作数据一致性？**

> EF Core 默认乐观并发，关键实体（如 DispatchOrder 状态变更）在 Service 层使用事务（`BeginTransactionAsync`）确保原子性。例如调度订单创建时，同一个事务内完成订单插入、合同创建、申请状态更新三步，避免半完成状态。

**Q：文件上传的安全性如何保证？**

> 五层校验：① 扩展名白名单（.jpg/.png/.pdf）；② Content-Type 校验；③ 文件魔数（前几字节）校验，防止伪造扩展名；④ 大小限制（10 MB）；⑤ 保存时使用 GUID 重命名，杜绝路径穿越。文件保存在 `Uploads/`（非 `wwwroot`），必须经过 `FilesController` 鉴权后才能访问。

**Q：角色权限如何扩展？**

> 角色名定义在 `Constants/Roles.cs` 静态类中，所有 Controller 注解引用常量而非硬编码字符串。新增角色只需：① 在常量类添加常量；② 在 `Roles.All` 数组加入以完成自动种子；③ 在相关 Controller 的 `[Authorize]` 注解中加入新角色。

**Q：数据库是否考虑过性能优化？**

> 对高频查询字段（如 `DispatchOrder.Status`、`DispatchOrder.ActualStart/End`、`Qualification.ValidTo`）建立了复合索引；通知表对 `(RecipientId, IsRead)` 建联合索引，支持快速未读消息查询；列表查询均使用分页，避免全表扫描。

**Q：系统如何处理证件过期提醒？**

> 在新增/更新资质证书时，服务层自动检测 `ValidTo ≤ 今天+30天`，如满足则向所有 Admin 和 DeviceAdmin 角色用户推送站内消息提醒。调度时也会过滤资质已过期的设备，从根本上阻止过期设备上线。
