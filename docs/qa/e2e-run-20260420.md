## 缺陷清单

### 🔴 P1 — 链路不贯通

#### Bug #1（残留于上一轮）：合同 PDF 导出后订单状态未推进
- **复现**：Suite 4c；Dispatcher 在 Order #6 详情页点"导出 PDF"生成 `CT-20260420-72FDC4.pdf`，合同文档生成，但 Order 仍为 `Unsigned` (Status=0)
- **后果**：Order 详情页"进场核验码"区块因 `@if (Model.OrderStatus >= DispatchOrderStatus.Signed)` 条件不满足，VerifyCode 对 PL/Dispatcher **完全不可见**，链路无法从"调度"流转到"核验"
- **Workaround 本次测试**：通过 `UPDATE DispatchOrders SET Status=1 WHERE Id=6` 直接推进
- **根因推测**：合同导出路径未把 Order 状态 `Unsigned → Signed`。要么需要上传已签扫描件路径，要么 demo-guide §4 少了"上传扫描件"一步
- **建议**：在 `Views/Dispatch/OrderDetails.cshtml` 的未上传签约扫描件时的 `Unsigned` 分支，显式提示"请上传签署扫描件"；或在合同详情页补"合同已签署"按钮跳过扫描件

### 🟡 P2 — 界面一致性

#### Bug #2：Chart.js race 不止影响首页
- **复现**：首页 `/`、`/Report/Utilization`、`/Report/Safety` 的浏览器控制台均出现 `ReferenceError: Chart is not defined` + `Unexpected token '<'`
- **计划标注**仅把首页列为已知；实际**三个页面**都踩坑
- **根因**：Chart.js 脚本通过 CDN + defer 加载，与内联 `new Chart(...)` 执行顺序竞争
- **建议**：把 Chart.js 引用移至 layout `<head>` 同步加载，或用 `DOMContentLoaded` + Chart 存在性检查包裹每段初始化；`_Layout.cshtml` 里修一处即可一次性解决三页

### 🟢 P3 — UX 细节

#### Bug #3：Audit Review 驳回不填备注时错误消息未渲染
- **复现**：Suite 3 反向；Auditor 进审核页点"驳回"但不填备注，服务端返回 400 但 `asp-validation-for="Form.Remark"` 位置并未显示 "驳回时必须填写原因"
- **原因**：`AuditController.cs:L32` 使用 `ModelState.AddModelError(nameof(form.Remark), ...)`，产出的 key = `Remark`；而 `Views/Audit/Review.cshtml` 绑定为 `Form.Remark`
- **修复**：`AuditController.cs` 改为 `ModelState.AddModelError("Form.Remark", "驳回时必须填写原因")`

#### Bug #4：Return Evaluate 扣款超押金无明确提示
- **复现**：Suite 8b；扣款 9000 > 押金 8000，系统**静默接受**并把退款钳位到 `¥0.00`，不给任何"扣款已超押金"提示
- **现状**：功能不错——不产生负退款——但 `Deduction=9000` **被写入 DB**，与 `Refund=0` 矛盾
- **建议**：在 `ReturnService.EvaluateAsync` 加 `if (form.Deduction > depositAmount) ModelState.AddError(...)` 或前端 JS 同步校验

#### Bug #5：Rental Excel 文件略小于计划预期
- **现状**：计划断言 ">5KB"，实际 3.3KB
- **原因**：测试期只有 2-3 条有效订单，行数少
- **结论**：非 bug，调整计划断言即可；或在种子数据里多放几单
