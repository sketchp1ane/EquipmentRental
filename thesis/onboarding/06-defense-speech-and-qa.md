## 三、逐页演讲稿

### 第 1 页：封面

各位老师好，我的毕业设计题目是《基于 ASP.NET Core MVC 的建筑租赁设备全生命周期管理平台》。本系统面向建筑设备租赁企业和施工项目现场，基于 ASP.NET Core MVC、EF Core 和 SQL Server 实现，主要解决设备租赁流程中台账分散、合同流转慢、安全记录难追溯等问题。

本次汇报我会从研究背景、系统设计、核心功能、安全测试和总结展望几个方面展开。

**过渡句**：首先看一下这个选题要解决的实际问题。

### 第 2 页：研究背景与问题

在建筑设备租赁场景中，传统管理方式经常依赖纸质台账或 Excel 表格。这样会带来三个问题：第一，设备状态不透明，比如一台设备到底是空闲、出租中还是维修中，更新不及时；第二，合同、调度、进场核验之间主要靠人工衔接，容易出现流程断点；第三，安全交底、巡检、故障和退场评价记录分散，后续追责和合规检查比较困难。

所以本项目的出发点不是单独做一个设备列表，而是希望把租赁全流程串起来。

**过渡句**：基于这些问题，我确定了系统设计的四个目标。

### 第 3 页：设计目标

本系统的目标可以概括为四点。第一是全生命周期管理，覆盖从设备入库到退场评价的完整链路。第二是角色化协同，不同岗位只能处理自己职责范围内的业务。第三是线上状态流转，合同上传、核验通过、故障处理、退场评价都会驱动业务状态变化。第四是安全与审计，包括权限控制、文件上传校验、富文本过滤和操作日志。

简单来说，就是让设备从“入库”到“退场”的每一步都有责任角色、业务凭证、状态推进和可追溯记录。

**过渡句**：接下来我用一张流程图说明系统的业务主线。

### 第 4 页：业务流程总览

系统主流程分为七个阶段：设备入库、资质审核、线上调度、进场核验、安全交底、使用监管和退场评价。

设备入库后默认是待审核状态，只有审核通过后才变为空闲并允许调度；调度阶段生成调度单和合同；合同扫描件上传后，订单进入已签署状态，才允许进场核验；核验通过后进入进行中；现场阶段可以做安全交底、巡检和故障上报；项目结束后提交退场申请，由设备管理员评价设备并计算押金退款。

这条流程体现了本系统最核心的设计：状态不是人工随便改，而是由业务动作推动。

**过渡句**：为了让不同业务人员按职责参与流程，系统设计了角色权限。

### 第 5 页：角色与权限设计

系统主要有五类角色：系统管理员、设备管理员、调度员、项目负责人和安全员。

系统管理员负责用户管理和全局查看；设备管理员负责设备入库、资质审核、故障处理和退场评价；调度员负责用车申请处理、排期和合同；项目负责人负责提交用车申请、进场核验和退场申请；安全员负责安全交底、巡检和故障上报。

这里需要说明的是，菜单隐藏只是前端体验层，真正的权限边界在后端 Controller 和 Action 上的 `[Authorize]` 注解。即使用户手动输入 URL，没有权限也会被后端拦截。

**过渡句**：权限解决了谁能做，接下来说明系统整体怎么组织代码和业务规则。

### 第 6 页：系统总体架构

系统采用 ASP.NET Core MVC 分层架构。View 层使用 Razor、Bootstrap、jQuery、Chart.js 和 Summernote 完成页面展示和交互。Controller 层只负责接收请求、权限控制和调用服务。核心业务规则都放在 Service 层，比如调度、合同、核验、故障和退场评价的状态流转。数据层使用 EF Core Code First 和 SQL Server。

这样设计的好处是 Controller 比较薄，业务逻辑集中在 Service，后续维护和测试会更清晰。

**过渡句**：在数据设计上，系统围绕设备和调度单串起主业务对象。

### 第 7 页：数据库设计

数据库的核心对象是 `Equipment` 和 `DispatchOrder`。设备表关联设备证件、审核记录、巡检记录和故障记录；调度单关联用车申请、合同、进场核验、安全交底和退场评价。

其中，合同和调度单是一对一关系，进场核验和调度单也是一对一关系。这样每一笔租赁业务都可以从设备追溯到合同、核验、安全记录和退场评价。

**过渡句**：下面按业务链路介绍几个核心功能。

### 第 8 页：设备台账与资质审核

第一个核心功能是设备台账和资质审核。设备管理员录入设备时，需要填写设备编号、分类、品牌型号、技术参数等信息，并可以上传图片。新设备创建后默认是待审核状态。

设备相关证件会在资质模块维护，例如产品合格证、年检报告和保险凭证。审核通过后，设备状态从待审核变为空闲；如果证件过期或未审核，该设备不会进入调度员可选列表。这保证了设备上线前先合规。

**过渡句**：设备合规后，就进入用车申请、调度和合同阶段。

### 第 9 页：线上调度与合同

项目负责人提交用车申请后，调度员根据设备分类、租赁时间和设备状态进行排期。系统在服务层过滤不可用设备，包括非空闲设备、证件过期设备和时间冲突设备。

排期后系统生成调度单和合同草稿，合同可以在线预览和导出 PDF。这里最关键的联动是：上传合同扫描件后，服务层会同时把合同状态改为已签署，把调度单状态改为已签署，并把设备状态改为使用中。这一步避免了合同已签但订单无法进入核验的流程断点。

**过渡句**：合同签署后，现场就可以进行进场核验和安全交底。

### 第 10 页：进场核验与安全交底

进场核验使用调度单绑定的 UUID 核验码。核验时服务层会检查订单是否已签署、设备是否处于使用中、证件是否有效，并限制核验码在租赁开始日期后 3 天内有效。核验成功后写入 `EntryVerification` 记录，并把调度单推进到进行中。

安全交底模块支持富文本内容、附件上传、参与人信息和签署确认。完成签署后，交底状态从草稿变为已完成，并可导出 PDF。这样现场安全管理也能形成可追溯记录。

**过渡句**：设备进入使用阶段后，系统还继续管理巡检、故障和退场。

### 第 11 页：巡检故障与退场评价

使用监管阶段主要包括日常巡检和故障上报。巡检使用固定检查项，任一项目异常时整体状态会标记为异常。故障上报后，设备状态会进入维修中，避免该设备被重复调度。

项目结束后，项目负责人提交退场申请，设备管理员填写退场评价，包括外观评分、功能评分、损耗描述和扣款金额。退款金额由服务端根据押金和扣款计算，不信任前端传值，避免篡改。

**过渡句**：除了业务流程，系统也做了安全与可靠性设计。

### 第 12 页：安全与可靠性设计

安全设计主要有六个方面。第一，认证使用 ASP.NET Core Identity，并替换为 BCrypt 密码哈希。第二，权限通过 `[Authorize]` 控制。第三，全局启用 CSRF 防护。第四，文件上传做扩展名、大小、MIME 类型、文件魔数和 GUID 重命名五层校验。第五，富文本内容入库前使用 HtmlSanitizer 过滤，防止 XSS。第六，关键操作写入审计日志。

另外，上传文件不放在 `wwwroot`，而是放在 `Uploads` 目录，访问时通过文件控制器鉴权下发，避免绕过权限直接访问文件。

**过渡句**：最后看一下系统验证情况。

### 第 13 页：测试与验证

测试方面，我使用真实浏览器进行端到端回归，覆盖登录、角色边界、设备入库、证件管理、资质审核、调度排期、合同签署、进场核验、安全交底、巡检故障和退场评价等主链路。

QA 记录显示主流程已经完整跑通。同时在回归过程中也发现并修复过一些问题，比如合同扫描件上传后的状态联动、Chart.js 加载顺序、字段级错误提示和退场扣款校验。这说明系统不是只停留在页面展示，而是经过了业务链路验证。

**过渡句**：最后做一个总结。

### 第 14 页：总结与展望

综上，本系统完成了建筑租赁设备从入库到退场的闭环管理，实现了设备可查、流程可控、风险可追溯。

当然，目前系统主要面向 Web 管理端，后续还可以继续扩展移动端扫码、GPS 或 IoT 设备状态采集、电子签章、财务发票和第三方 ERP 对接等能力。

我的汇报结束，请各位老师批评指正。

## 四、答辩问答：短答版 + 展开版

### 1. 你的项目创新点在哪里？

**短答版**：创新点在业务闭环和状态联动，不是单独做几个 CRUD 页面。

**展开版**：系统把设备状态、资质审核、合同签署、进场核验、安全交底、故障处理和退场评价串成一条完整状态链路，并结合五类角色权限和操作日志，实现“谁操作、什么状态、有什么凭证、是否可追溯”都能在系统中体现。

### 2. 为什么选择 ASP.NET Core MVC，而不是前后端分离？

**短答版**：因为这是企业内部管理系统，表单多、权限强、开发周期有限，MVC 更适合快速完成业务闭环。

**展开版**：ASP.NET Core MVC 和 Identity、DataAnnotations、Razor 表单验证集成度比较高，适合后台管理系统。前后端分离适合交互复杂、多端复用或 C 端高并发场景，但本项目第一版更重视流程完整、权限安全和开发效率，所以选择 MVC 更合适。

### 3. 你的系统如何保证不同角色不能越权？

**短答版**：菜单隐藏只是辅助，真正权限控制在后端 `[Authorize]`。

**展开版**：系统使用 ASP.NET Core Identity 做认证授权，角色名统一定义在 `Constants/Roles.cs`。各 Controller 或 Action 使用 `[Authorize(Roles=...)]` 限制访问。即使用户手动输入 URL，后端仍会判断角色，不符合要求就拒绝访问。

### 4. 数据库设计的核心是什么？

**短答版**：核心是以 `Equipment` 和 `DispatchOrder` 串起全流程。

**展开版**：设备关联证件、审核、巡检和故障；调度单关联合同、核验、安全交底和退场评价。合同与调度单 1:1，核验记录与调度单 1:1，这样每一笔租赁业务都能从设备追溯到合同、现场记录和退场结算。

### 5. 如何避免同一设备被重复调度？

**短答版**：服务层会过滤状态、证件和时间冲突。

**展开版**：调度员选择设备时，系统只展示状态为空闲、证件未过期、时间段无冲突的设备。创建调度单时服务层还会再次校验，不只依赖前端列表，避免因页面数据过期导致重复调度。

### 6. 合同上传为什么会影响设备状态？

**短答版**：因为合同扫描件是系统认定“线下签署完成”的业务凭证。

**展开版**：本项目暂不做电子签章，所以采用线下签署后上传扫描件的方式。上传成功后，服务层同时把 `Contract` 改为 `Signed`，把 `DispatchOrder` 改为 `Signed`，把 `Equipment` 改为 `InUse`。这样后续进场核验才有明确前置条件。

### 7. 核验码怎么保证有效性？

**短答版**：核验码是 GUID，和调度单绑定，并且服务端校验状态和有效期。

**展开版**：核验时系统检查四点：核验码是否存在、订单是否已签署、设备是否处于使用中、设备证件是否过期。同时核验码限制在租赁开始日期后 3 天内有效。通过后写入核验记录并推进订单状态。

### 8. 文件上传怎么保证安全？

**短答版**：扩展名、大小、MIME、魔数、GUID 重命名五层校验。

**展开版**：系统只允许 JPG/JPEG/PNG/PDF，单文件不超过 10 MB；同时检查 Content-Type 和文件头魔数，防止伪装扩展名；保存时使用 GUID 文件名，避免路径穿越和重名覆盖；文件放在 `Uploads` 而不是 `wwwroot`，下载时必须经过鉴权。

### 9. 如何防止 XSS、CSRF 和 SQL 注入？

**短答版**：富文本用 HtmlSanitizer，POST 用全局 CSRF，数据库访问用 EF Core LINQ。

**展开版**：安全交底中有富文本输入，所以保存前用 HtmlSanitizer 清洗危险标签。项目在全局启用 antiforgery token 校验，防止跨站请求伪造。数据库操作使用 EF Core LINQ，不拼接原始 SQL，从常见风险入口做防护。

### 10. 并发和事务怎么处理？

**短答版**：关键状态变化集中在 Service 层，创建调度单时使用事务。

**展开版**：例如创建调度单时，同一个事务内完成调度单插入、合同创建和申请状态更新，避免出现只生成订单但没有合同的半完成状态。后续如果要支持更高并发，可以为关键实体增加 `RowVersion` 乐观并发字段。

### 11. 你做了哪些测试？

**短答版**：做了真实浏览器端到端回归，主链路已经跑通。

**展开版**：测试覆盖登录、角色边界、设备入库、证件、审核、调度、合同、核验、安全交底、巡检、故障和退场评价。QA 文档中记录了完整业务链路，也记录并修复过合同状态联动、Chart.js 加载、字段级错误提示和扣款校验等问题。

### 12. 系统目前有什么不足？

**短答版**：当前主要是 Web 管理端，移动端、IoT 和电子签章还没有接入。

**展开版**：现在系统优先保证租赁主流程闭环和安全边界。后续可以扩展移动端扫码核验、GPS/IoT 设备状态采集、电子签章、财务发票系统和第三方 ERP 对接。

### 13. 为什么上传文件不直接放在 `wwwroot`？

**短答版**：避免用户绕过权限直接访问文件。

**展开版**：`wwwroot` 是静态资源目录，如果上传文件直接放进去，知道路径的人可能绕过业务权限访问。系统把文件放在 `Uploads`，再通过 `FilesController` 做鉴权下发，这样能和系统权限保持一致。

### 14. 为什么要把业务逻辑放在 Service，而不是 Controller？

**短答版**：为了让 Controller 保持薄层，业务规则集中维护。

**展开版**：Controller 负责接收请求、校验权限和返回页面；Service 负责状态流转、事务和业务规则。比如合同上传影响合同、订单和设备三个对象，如果写在 Controller 里会很分散，放在 Service 里更容易维护和测试。

### 15. 设备证件过期时系统怎么处理？

**短答版**：首页预警，调度过滤，核验再校验。

**展开版**：证件临近到期会在首页看板展示预警，并通过通知提醒相关角色；调度选择设备时会过滤过期证件设备；进场核验时服务层还会再次检查证件有效期，防止过期设备进入现场。

### 16. 你的系统和普通设备管理系统有什么区别？

**短答版**：普通系统偏台账，本系统偏租赁流程闭环。

**展开版**：本系统不仅记录设备基本信息，还把设备资质、调度合同、进场核验、安全交底、巡检故障和退场评价串起来。它关注的是租赁过程中的状态、责任人、凭证和追溯。

### 17. 如果老师问“你觉得最难实现的是哪一块？”

**推荐回答**：最难的是跨模块状态联动。比如合同扫描件上传会同时影响合同、调度单和设备状态；进场核验又依赖订单状态、设备状态和证件有效期。为了避免逻辑分散，我把这些规则放在 Service 层统一处理。

### 18. 如果老师问“你收获最大的是哪一点？”

**推荐回答**：我最大的收获是理解了管理系统不只是页面增删改查，更重要的是业务状态和权限边界。比如同样是修改状态，如果没有明确前置条件和审计记录，就会造成流程不可控。

## 六、3 个临场备用版本

### 30 秒超短项目介绍

本项目是一个基于 ASP.NET Core MVC 的建筑租赁设备全生命周期管理平台，覆盖设备入库、资质审核、调度合同、进场核验、安全交底、巡检故障和退场评价七个阶段。系统通过五类角色权限、服务层状态流转、文件安全上传和端到端测试，实现了设备可查、流程可控、风险可追溯。

### 1 分钟技术介绍

系统采用 MVC 分层架构，View 层使用 Razor 和 Bootstrap，Controller 层负责请求和权限，Service 层负责业务规则、事务和状态流转，数据层使用 EF Core Code First 和 SQL Server。认证授权使用 ASP.NET Core Identity，文件上传做白名单、MIME、魔数和大小校验，富文本使用 HtmlSanitizer 过滤，关键操作写入审计日志。

### 1 分钟功能介绍

业务上，设备管理员先录入设备并维护证件，审核通过后设备才能被调度。项目负责人提交用车申请，调度员排期并生成合同。合同扫描件上传后，系统推进合同、订单和设备状态，然后项目负责人进行进场核验。现场阶段可以做安全交底、巡检和故障处理，最后通过退场评价完成押金结算和设备状态回写。
