# 05 业务流程写作素材

## 总体流程

系统围绕建筑租赁设备的全生命周期展开，主流程如下：

```text
设备入库 -> 资质审核 -> 线上调度 -> 进场核验 -> 安全交底 -> 使用监管 -> 退场评价
```

论文中应强调：这些阶段不是孤立页面，而是通过状态流转、角色权限和数据关联构成业务闭环。

## 阶段说明

| 阶段 | 操作角色 | 输入 | 输出 | 状态变化 |
|---|---|---|---|---|
| 设备入库 | 设备管理员、系统管理员 | 设备编号、分类、型号、图片、所属公司 | 设备记录、设备图片 | 设备进入待审核 |
| 资质维护 | 设备管理员、系统管理员 | 合格证、年检报告、保险等 | 证件记录 | 为审核和调度提供依据 |
| 资质审核 | 设备管理员、系统管理员 | 审核动作、备注 | 审核记录、通知 | 审核通过后设备变为空闲 |
| 用车申请 | 项目负责人、系统管理员 | 项目、地址、设备类型、日期、联系人 | 用车申请 | 申请进入待处理 |
| 调度排期 | 调度员、系统管理员 | 设备、租期、日租金、押金 | 调度单、合同草稿 | 申请变为已排期 |
| 合同签署 | 调度员、系统管理员 | 合同扫描件 | 已签署合同 | 合同和调度单变为已签署，设备进入出租中 |
| 进场核验 | 项目负责人、系统管理员 | 核验码 | 核验记录 | 调度单进入进行中，设备保持出租中 |
| 安全交底 | 安全员、系统管理员 | 富文本内容、参与人、附件 | 交底记录、签署记录 | 交底从草稿到已完成 |
| 日常巡检 | 安全员、系统管理员 | 检查项、整体状态、备注、照片 | 巡检记录 | 使用过程形成检查留痕 |
| 故障处理 | 安全员、项目负责人、设备管理员、系统管理员 | 故障描述、维修结果 | 故障工单 | 安全员/项目负责人/管理员上报故障；设备管理员/管理员受理和关闭故障 |
| 退场评价 | 项目负责人、设备管理员 | 退场申请、评分、扣款、状态 | 退场评价、押金结果 | 调度单完成，设备回到后续状态 |

## 设备状态线

```text
PendingReview 待审核
  -> Idle 空闲
  -> InUse 出租中
  -> Maintenance 维修中
  -> Idle 空闲 / Maintenance 维修中 / Scrapped 已报废
```

写作说明：

- 新建设备默认不是可租赁状态，而是待审核。
- 审核通过后设备进入空闲状态，才可能被调度。
- 当前主流程在上传合同扫描件时推动设备进入出租中；进场核验校验设备已出租中并推进调度单。
- 故障处理和退场评价会影响设备后续状态。

## 调度和合同状态线

```text
DispatchRequest: Pending 待处理 -> Scheduled 已排期 / Cancelled 已取消
DispatchOrder: Unsigned 待签署 -> Signed 已签署 -> InProgress 进行中 -> Complete 已完成
Contract: Draft 草稿 -> Signed 已签署
```

写作说明：

- 用车申请由项目负责人提交，调度员处理。
- 调度单生成后合同处于草稿状态。
- 合同扫描件上传后，合同和调度单才进入已签署状态，并同步把设备置为出租中。
- 进场核验必须基于已签署调度单。
- `ContractStatus.AwaitingSignature` 是枚举保留状态，当前主流程未实际写入。

## 安全与故障状态线

```text
SafetyBriefing: Draft 草稿 -> Completed 已完成
FaultReport: Pending 待处理 -> InProgress 处理中 -> Closed 已关闭
ReturnApplication: PendingEvaluation 待评价 -> Complete 已完成
```

写作说明：

- 安全交底用于记录现场交底内容、参与人签署和附件。
- 故障工单用于记录从上报到维修关闭的全过程。
- 退场评价用于沉淀设备状态、损耗扣款和押金退还结果。

## 可直接写入论文的段落

系统以设备生命周期为主线，将设备入库、资质审核、调度租赁、合同签署、进场核验、安全交底、使用监管和退场评价等环节串联起来。每个环节由不同角色执行，并在数据库中形成对应业务记录。系统通过状态字段描述设备、用车申请、调度单、合同、故障和退场申请的当前阶段，使业务进展能够被追踪和约束。与传统设备台账管理相比，该设计不仅记录设备基本信息，还能够反映设备在租赁过程中的业务状态和责任流转。
