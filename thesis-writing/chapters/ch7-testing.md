# 7 系统测试

## 7.1 测试环境

系统测试在本地开发环境中进行，主要验证系统功能、角色权限、业务状态流转和安全机制是否符合预期。

## 7.2 测试方法与测试内容

软件测试是保证系统质量的重要环节。Pressman 指出，测试应覆盖功能正确性、边界条件和异常情况，以发现系统实现与需求之间的偏差[9]。本系统测试主要采用黑盒测试和业务流程测试方法，从用户操作角度验证系统能否完成预期功能。

测试内容包括四个方面。第一，功能测试，验证用户登录、设备入库、资质审核、调度排期、合同上传、进场核验、安全交底、故障处理和退场评价等功能是否正常。第二，权限测试，验证不同角色是否只能访问与其职责相关的页面和操作。第三，业务流程测试，按照设备租赁全生命周期流程从设备入库执行到退场评价，检查状态是否正确流转。第四，安全性测试，验证非法文件上传、缺少 CSRF 令牌、富文本危险脚本和越权访问是否能够被系统拦截或限制。

## 7.4 权限与安全性测试

权限测试主要验证五类角色之间的访问边界。例如，未登录用户访问受保护页面时应跳转到登录页面；安全员访问用户管理页面应被拒绝；项目负责人不能执行调度排期；非调度员或系统管理员不能上传合同扫描件；非设备管理员或系统管理员不能进行资质审核、故障关闭和退场评价。测试结果表明，系统能够根据用户角色限制操作入口和请求访问，基本满足角色权限隔离要求。

安全性测试主要包括文件上传校验、CSRF 防护和富文本过滤。文件上传测试覆盖非法扩展名、伪造扩展名导致魔数不符、超过 10 MB 文件和上传路径直接访问等场景。系统对修改类请求启用防伪令牌校验，降低跨站请求伪造风险[17]。安全交底富文本内容在保存前进行过滤，减少 XSS 风险[19]。测试结果表明，系统能够对常见输入和文件上传风险进行基本防护。

## 7.5 业务流程测试

业务流程测试按照以下顺序执行：

```text
设备入库 -> 添加证件 -> 资质审核通过 -> 提交用车申请 -> 调度排期
-> 上传合同扫描件 -> 进场核验 -> 安全交底 -> 巡检/故障处理
-> 退场申请 -> 退场评价
```

测试过程中重点检查每一步是否满足前置条件，以及状态是否正确推进。例如，未审核设备不能进入可调度状态；调度排期后生成待签署调度单和合同草稿；合同扫描件上传后，合同状态和调度单状态变为已签署，设备状态变为出租中或使用中；进场核验通过后，系统生成核验记录并将调度单推进为进行中；退场评价提交后，调度单应变为已完成，设备状态根据评价结果更新为空闲、维修中或已报废。测试结果表明，系统主要业务状态能够按照预期流转。

## 7.6 测试结果分析

通过功能测试、权限测试、业务流程测试和安全性测试可以看出，系统已实现建筑租赁设备全生命周期管理的主要功能。测试结果表明，系统能够完成设备入库、资质审核、线上调度、合同管理、进场核验、安全交底、巡检监管、故障处理和退场评价等核心流程；不同角色之间的权限边界较为清晰；文件上传和富文本输入具备基本安全防护能力。由于系统主要在本地环境中进行测试，尚未进行大规模并发和真实生产环境验证，后续仍需进一步完善性能测试和部署运维方案。
