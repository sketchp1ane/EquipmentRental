# 3 系统需求分析

## 3.1 系统可行性分析

从技术可行性看，ASP.NET Core MVC、EF Core 和 SQL Server 均为成熟的软件开发技术，能够满足 B/S 管理系统的开发需求。系统所需的用户认证、角色授权、表单提交、数据查询、文件上传、图表展示和文档导出均可以通过现有框架和组件实现。项目采用分层架构组织代码，便于后续维护和扩展。

从经济可行性看，系统主要面向毕业设计和中小规模建筑租赁设备管理场景，开发过程主要使用常见开发工具和开源或社区可用组件。系统通过信息化方式替代部分纸质台账和线下流转过程，可以减少重复登记和人工核对工作，提高设备状态查询、合同流转和安全记录追溯效率。

从操作可行性看，系统采用浏览器访问方式，不需要用户安装专用客户端。系统按照角色划分菜单和操作入口，使设备管理员、调度员、项目负责人和安全员能够根据自身职责完成业务操作。页面采用表单、列表、详情和看板等常见交互方式，符合管理系统用户的使用习惯。

## 3.2 用户角色分析

系统涉及六类用户角色。不同角色拥有不同的业务职责和操作权限，系统通过角色授权实现权限隔离。

表3.1 用户角色职责表

| 角色 | 主要职责 | 典型操作 |
|---|---|---|
| 系统管理员 | 负责全局管理、用户管理和完整业务演示 | 新增用户、分配角色、启停账号、执行业务演示 |
| 设备管理员 | 负责设备资产、资质证件、故障处理和退场评价 | 设备入库、证件维护、资质审核、维修处理、退场评价 |
| 调度员 | 负责用车申请处理、设备排期和合同管理 | 处理申请、筛选可用设备、生成合同、上传合同扫描件 |
| 项目负责人 | 代表项目提交设备使用需求并完成进退场操作 | 提交用车申请、进场核验、发起退场申请 |
| 安全员 | 负责设备使用过程中的安全管理 | 创建安全交底、巡检记录、故障上报 |
| 只读审计员 | 负责查看被授权页面和记录，验证权限隔离 | 查看授权范围内记录，不执行审核、调度、上传、关闭或评价等修改操作 |

通过角色划分，系统能够将设备管理、调度合同、安全监管和项目使用等职责分离，减少越权操作和职责混淆。例如，只读审计员只能查看其被授权的页面和记录，不能进行资质审核、调度排期、合同扫描件上传、巡检创建、故障关闭或退场评价等修改类操作。

## 3.3 功能性需求分析

系统功能围绕建筑租赁设备全生命周期展开，主要包括用户与认证、设备台账、资质审核、线上调度、合同管理、进场核验、安全交底、使用监管、故障处理、退场评价、首页看板、站内消息和文件访问等模块。

表3.2 系统核心功能需求表

| 模块 | 功能需求 | 说明 |
|---|---|---|
| 用户与认证 | 登录、注销、个人信息、密码修改、用户管理 | 支持用户启停、角色分配和权限控制 |
| 设备台账 | 设备分类基础字典、设备入库、图片上传、列表筛选、详情查看、Excel 导出 | 分类用于入库选择、申请选择和列表筛选；新建设备进入待审核状态 |
| 资质审核 | 证件维护、到期预警、审核通过、审核驳回 | 审核通过后设备可进入空闲状态 |
| 线上调度 | 用车申请、可用设备筛选、调度排期、调度日历 | 根据设备状态、资质和租期筛选可用设备 |
| 合同管理 | 合同草稿、在线预览、PDF 导出、扫描件上传 | 上传合同扫描件后推进合同、调度单和设备状态 |
| 进场核验 | 核验码展示、核验结果记录、状态推进 | 已签署调度单才能执行进场核验 |
| 安全交底 | 富文本交底、参与人签署、附件、PDF 导出 | 保存现场安全交底过程资料 |
| 使用监管 | 巡检记录、固定巡检项、现场照片 | 记录设备使用过程中的检查情况 |
| 故障处理 | 故障上报、工单接受、维修处理、关闭恢复 | 故障可能影响设备状态 |
| 退场评价 | 退场申请、评分、损耗扣款、押金退还、设备状态更新 | 完成租赁流程闭环 |
| 首页看板 | 设备统计、租赁趋势、证件预警、角色待办 | 为不同角色提供业务提醒 |
| 站内消息 | 通知提醒、未读消息、跳转链接 | 支持跨角色协同 |
| 文件访问 | 文件上传安全校验、受控下载 | 上传文件不直接暴露在静态目录下 |

## 3.4 非功能性需求分析

除功能需求外，系统还需要满足安全性、可维护性、易用性、数据一致性和可追溯性等非功能需求。ISO/IEC 25010:2011 将软件质量划分为功能适合性、性能效率、兼容性、易用性、可靠性、安全性、可维护性和可移植性等方面[7]，这些指标对本系统同样具有参考意义。

在安全性方面，系统需要对用户身份和角色权限进行校验，防止未登录访问和越权操作；对文件上传进行扩展名、MIME 类型、文件魔数和大小限制校验，降低恶意文件上传风险；对富文本内容进行过滤，防止危险脚本进入系统。在可维护性方面，系统采用 MVC 分层架构和 Service 层业务逻辑集中管理，使控制器、服务、实体、视图模型和页面之间职责清晰。在易用性方面，系统采用常见的表单、列表、详情和看板页面，使用户能够快速完成业务操作。在数据一致性方面，系统通过状态字段和服务层规则约束业务流程，避免未审核设备被调度、未签署合同直接核验等问题。在可追溯性方面，系统保存审核记录、合同记录、核验记录、安全交底、巡检、故障和退场评价等过程数据，为责任追踪提供依据。

## 3.5 核心业务流程分析

系统核心流程如下：

```text
设备入库 -> 资质审核 -> 线上调度 -> 进场核验 -> 安全交底 -> 使用监管 -> 退场评价
```

设备管理员首先录入设备基本信息和图片，设备进入待审核状态。随后，设备管理员维护设备资质证件并进行资质审核，审核通过后设备进入空闲状态。项目负责人根据项目需求提交用车申请，调度员根据设备类型、租期、设备状态和资质有效期筛选可用设备并生成调度单和合同草稿。合同线下签署后，调度员或管理员上传合同扫描件，系统将合同置为已签署、调度单置为已签署，并将设备状态置为使用中。项目负责人随后通过核验码或二维码完成进场核验，核验通过后系统生成进场核验记录，并将调度单推进为进行中。设备使用过程中，安全员或管理员创建巡检记录，安全员、项目负责人或管理员可以上报故障，设备管理员或管理员负责故障处理和关闭。租赁结束后，项目负责人发起退场申请，设备管理员进行退场评价，记录评分、损耗扣款和押金退还结果，最终完成设备租赁业务闭环。
