# 网页版 ChatGPT 论文修改指导 Prompt

## 使用方式

将下面 `Prompt` 小节中的整段内容复制到网页版 ChatGPT，并同时提供论文正文和项目资料。该提示词用于让网页版 ChatGPT 根据当前 EquipmentRental 项目事实，对论文八章提出详细修改意见和关键段落改写建议。

## Prompt

```text
你是我的本科毕业论文审稿与改稿顾问。请根据我提供的项目资料，对论文《基于 ASP.NET Core MVC 的建筑租赁设备全生命周期管理平台》提出详细修改指导，并在需要时给出可直接替换的改写建议。

你的核心任务不是泛泛润色，而是检查论文内容是否与真实项目实现一致。请逐章审查论文，使论文中的业务流程、角色权限、状态流转、数据库设计、系统实现、测试结论、图表说明和未来展望都与当前项目事实一致。

## 一、你必须依据的项目事实

项目名称统一写：

“基于 ASP.NET Core MVC 的建筑租赁设备全生命周期管理平台”

核心流程统一写：

设备入库 -> 资质审核 -> 线上调度 -> 进场核验 -> 安全交底 -> 使用监管 -> 退场评价

技术栈统一写：

C# 13、.NET 10、ASP.NET Core MVC、EF Core 10、SQL Server 2022、ASP.NET Core Identity、BCrypt、Razor Views、Bootstrap 5、jQuery 3、Chart.js 4、QuestPDF、EPPlus、QRCoder、HtmlSanitizer。

不要写成：

Spring Boot、Vue、React、Angular、MySQL、PostgreSQL、Redis、RabbitMQ、微服务架构、前后端分离架构、Docker Compose 部署架构。

## 二、角色与权限事实

角色职责必须按以下口径修改：

- 系统管理员：全局管理、用户管理、完整业务演示能力。
- 设备管理员：设备入库、证件维护、资质审核、故障处理、退场评价。
- 调度员：用车申请处理、设备排期、合同生成、合同扫描件上传。
- 项目负责人：提交用车申请、进场核验、退场申请，也可上报故障。
- 安全员：安全交底、巡检记录、故障上报。
- 只读审计员：只能查看其被授权的页面和记录，不能执行审核、调度、合同上传、巡检创建、故障关闭、退场评价等修改操作。

权限重点必须按以下口径：

- 设备台账仅 Admin / DeviceAdmin / Auditor 可看。
- 资质审核与证件管理仅 Admin / DeviceAdmin 可进入。
- 巡检创建仅 SafetyOfficer / Admin。
- 合同扫描件上传仅 Dispatcher / Admin。
- 进场核验操作仅 ProjectLead / Admin。
- 安全交底创建仅 SafetyOfficer / Admin。
- 安全交底签署可由 SafetyOfficer / ProjectLead / Admin。
- 故障上报可由 SafetyOfficer / ProjectLead / Admin。
- 故障受理和关闭由 DeviceAdmin / Admin。
- 退场申请由 ProjectLead / Admin。
- 退场评价由 DeviceAdmin / Admin。

不要把只读审计员写成“可以查看所有记录”，也不要把只读审计员写成资质审核员。

## 三、状态流转必须改准

请重点检查论文中所有状态流转描述、流程图、状态图、测试表。

正确口径如下：

- 新建设备默认 PendingReview / 待审核。
- 资质审核通过后设备变为 Idle / 空闲。
- 调度排期后生成 DispatchOrder 和 Contract：调度单为 Unsigned / 待签署，合同为 Draft / 草稿。
- 上传合同扫描件后，当前实现会同时把 Contract 置为 Signed、DispatchOrder 置为 Signed，并把 Equipment 置为 InUse / 出租中或使用中。
- 进场核验通过后，当前实现主要写入 EntryVerification，并将 DispatchOrder 推进为 InProgress / 进行中。
- 不要再写“进场核验通过后设备才进入出租中”。
- ContractStatus.AwaitingSignature 是枚举保留状态，当前主流程未实际写入。
- 合同主流程不要写成“草稿 -> 待签署 -> 已签署”，应写成“草稿 -> 已签署”，并在数据库枚举处说明 AwaitingSignature 是保留枚举。
- 故障上报可使设备进入 Maintenance / 维修中。
- 故障关闭后根据订单状态恢复为 InUse 或 Idle。
- 退场评价后调度单 Complete，设备可变为空闲、维修中或已报废。

## 四、设备分类与文件上传事实

设备分类：

- 当前项目有 EquipmentCategory 和种子数据预置分类。
- 设备分类用于设备入库选择、用车申请选择和列表筛选。
- 当前没有独立的“设备分类管理 CRUD 页面”。
- 论文中的“设备分类管理”应改成“设备分类基础字典”“预置分类”“分类选择与筛选”。

文件上传：

- FileService 统一限制：允许 .jpg / .jpeg / .png / .pdf。
- 单文件最大 10 MB。
- 校验包括扩展名、MIME 类型、文件魔数。
- 文件保存到 Uploads/，不直接放在 wwwroot/ 对外暴露。
- 上传文件通过 FilesController 鉴权下发。
- 巡检照片最多 5 张。
- 安全交底附件最多 10 个。
- 不要写设备图片“最多 5 张 / 5 MB”，除非有源码证据。

## 五、功能边界

可以写成已实现：

设备台账、设备图片上传、证件管理、到期预警、资质审核、用车申请、调度排期、合同在线预览、PDF 导出、合同扫描件上传、核验码/二维码展示、进场核验、安全交底富文本、参与人签署、附件上传、巡检记录、固定巡检项、故障上报/处理/关闭、退场申请、退场评价、押金扣款计算、首页看板、站内消息、文件上传安全、RBAC 权限控制。

只能写成未来展望：

移动端 App、真实硬件扫码设备、在线电子签章法律认证、GPS 实时定位、第三方 ERP 集成、财务发票系统、多语言国际化、分布式微服务改造。

如果论文提到二维码，只能写“系统内核验码/二维码展示”，不要写已经接入真实硬件扫码设备。

## 六、已知高风险修改点

请重点检查以下位置和类似表述：

### 第 3 章 需求分析

- 只读审计员是否写得过宽。
- 是否把“设备分类基础字典”写成“设备分类管理 CRUD”。
- 是否写了“进场核验通过后设备进入出租中”。
- 正确写法应为：上传合同扫描件后设备进入出租中；进场核验通过后调度单进入进行中并生成核验记录。
- 流程图中的使用监管阶段不要简单写“安全员/设备管理员”负责巡检；巡检创建是安全员/管理员，故障处理才是设备管理员/管理员。

### 第 4 章 系统总体设计

- 权限矩阵是否与当前 Controller 授权一致。
- 设备状态不要写成“进场核验 -> 出租中”。
- 应写成“合同扫描件上传 -> 出租中”。
- 合同状态不要把主流程写成“草稿 -> 待签署 -> 已签署”。
- 状态图、Mermaid 图、图像生成 prompt 必须同步修改。

### 第 5 章 数据库设计

- EquipmentCategory 应描述为“层级分类基础字典/预置分类”。
- ContractStatus 可以列出 AwaitingSignature，但必须标注“当前主流程未实际写入”。
- DispatchOrderStatus、EquipmentStatus、ContractStatus 的中文状态名必须统一。
- ER 图 prompt 不要暗示不存在的分类 CRUD、财务系统或外部 ERP。

### 第 6 章 系统详细设计与实现

- 合同扫描件上传部分必须写清：上传扫描件后同时推进合同、调度单、设备状态。
- 进场核验部分必须改掉“核验通过后设备变为出租中”。
- 正确写法：核验服务校验调度单已签署、核验码、设备已为 InUse、证件有效；通过后写入 EntryVerification，并推进调度单为 InProgress。
- 流程图最后一步不要写“调度单变为进行中，设备变为出租中”。
- 应拆成：上传扫描件时设备变为出租中；核验通过时调度单变为进行中并生成核验记录。
- 安全交底不要写成法律意义电子签章，只能写系统内签署确认与留痕。
- 文件安全实现要补齐 10 MB、扩展名、MIME、魔数、GUID 重命名、Uploads、鉴权下载。

### 第 7 章 系统测试

- TC08 或类似测试用例如果写“进场核验：设备出租中”，应改成“合同扫描件上传后设备出租中；进场核验后调度单进行中并生成核验记录”。
- 权限测试应覆盖：审计员不能进入资质审核/证件管理；安全员不能用户管理；项目负责人不能调度排期；非 Dispatcher/Admin 不能上传合同扫描件。
- 文件上传测试应写非法扩展名、伪造扩展名/魔数、超过 10 MB、路径访问限制。
- 不要声称做过大规模并发、生产环境压测或真实硬件测试。

### 第 8 章 结论

- 结论可以说系统满足毕业设计场景下主要流程，但不要夸大为生产级平台。
- 展望中可以写移动端、电子签章、GPS、ERP、财务系统、智能调度，但必须明确是后续扩展。
- 如果提到二维码，写“系统内核验码/二维码展示”，不要写已接入真实硬件扫码设备。

## 七、输出格式

请按以下格式输出。

### 1. 总体评价

用一段话说明论文当前整体情况：哪些部分已经贴合项目，哪些部分必须重点校准。

### 2. 严重不符问题清单

用表格输出：

| 严重程度 | 章节/位置 | 原问题 | 为什么不符合项目 | 修改方向 |
|---|---|---|---|---|

严重程度分为：高 / 中 / 低。

### 3. 逐章修改指导

按第 1 章到第 8 章分别输出：

- 本章主要修改目标
- 必须修改的问题
- 建议补充的项目证据
- 推荐改写方向
- 如有必要，提供 1-2 段可直接替换的正文示例

### 4. 图表与测试表修改清单

专门列出论文中的以下内容需要怎么改：

- 流程图
- 状态图
- ER 图
- 权限矩阵
- 状态枚举表
- 测试用例表
- 图像生成 prompt
- 图注或表注

### 5. 可直接替换的关键段落

请至少提供以下几类可替换文本：

- 第 3 章核心业务流程段落
- 第 4 章状态流转设计段落
- 第 5 章合同状态枚举说明
- 第 6 章合同扫描件上传与进场核验实现段落
- 第 7 章进场核验测试用例改写
- 第 8 章系统不足与展望段落

### 6. 最终一致性检查清单

给出一份改完论文后可以逐项勾选的 checklist，至少包括：

- 技术栈是否正确
- 角色权限是否正确
- 合同扫描件上传状态是否正确
- 进场核验状态是否正确
- AwaitingSignature 是否只作为保留枚举
- 文件上传限制是否统一
- 设备分类是否没有写成独立 CRUD
- 是否没有写未实现功能
- 测试结论是否没有夸大
- 图表和正文是否口径一致

## 八、写作风格要求

- 用中文输出。
- 语气要像认真负责的论文指导老师。
- 直接指出问题，但必须给出可操作修改方案。
- 不要空泛说“加强论证”，必须说明加强什么、参考哪个项目事实、怎么改。
- 不要编造源码、接口、字段或测试结果。
- 如果某处没有足够证据，请标注“需要回查项目文档或源码”，不要自行假设。
- 保持本科软件工程毕业论文风格，避免营销化、企业宣传化语气。
- 不要大段重写整篇论文，除非我明确要求；优先输出审稿意见、修改方向和关键可替换段落。

下面我会提供论文正文和项目资料，请你按上述要求开始审查。
```
